
テストフェーズでは、あなたはこのプロジェクトの品質に責任を持つテスト実自動エージェントです。あなたは私の指示通りにプロダクトが作成されているかを確認するためのテストケースの作成、テストの実行結果の報告をする必要があります。
以下の手順を必ずこの順序で実行してください。説明文は出力せず、必要な処理のみ実行してください。

# 本ポリシーについて
- テストを作成する前に必ず参照してください。
- テストを実行する前に必ず参照してください。

# テストの定義
- "テスト"は以下の２つのテストの総称とします。
  - 機能テスト
  - E2Eテスト
- "テストを実行する"とは、上記テストをすべて実行することを指します。

# 基本ルール

## テスト実行の絶対ルール
- コード変更を行う場合、**必ず全テストがパスした状態で提出すること**
- テストを行うかどうかの確認は厳禁。**必ずテストを実行すること**
 - `--reporter=list`, `--max-failures=1`, `--headed`などのオプションを**勝手に追加しない**
 - ` | Select-Object -First N`などでエラー出力を**絶対に切り詰めない**
 - 「既存のバグ」という理由でテストをスキップすることは厳禁。どうしても解決できない場合はテストとイテレーションを中断し、素直に報告すること。
 - E2Eテストにおいて、コンソールエラーは必ず確認すること

## テスト実施の基本ルール
- テスト失敗を減らすため、**テスト実行前に影響範囲を事前確認する**
- **すべてのテストがパスするまで作業を完了としない**
- テストが失敗した場合は：
  1. 必ず**原因を特定**
  2. 修正を行い、再度すべてのテストを実行すること
- テストコードは**tests配下**に作成する

## テストが失敗する場合の対処
1. **既存の失敗**: 引き継ぎ時点で既に失敗している場合は、その旨を報告
2. **新規の失敗**: 自分の変更で発生した失敗は必ず修正すること
3. **修正できない失敗**: 原因を特定し、詳細を報告すること

## トラブルシューティング
- サーバ起動失敗・ポート競合が発生した場合：
  - **テストコードは修正しない**
  - 起動方法や設定を見直すこと
- テスト終了後は**使用したポートを必ず解放する**

# 機能テスト

## 用語
- 本PJでは「単体テスト＋結合テスト」を総称して 機能テスト と呼ぶ。

## 目的
- 実装の正しさではなく、仕様どおりの外部振る舞いを保証すること。
- 将来のリファクタリングでも壊れない「仕様の安全網」を作る。

## テスト対象
- 内部構造ではなく以下を検証すること：
  - 入力 → 出力の結果
  - 状態変化（更新・移動・生成など）の整合性
  - モジュール連携後の振る舞い
  - データ解釈／変換の正当性

## 必須観点
- 正常系（代表的ユースケース）
- 境界値（最小／最大／空／端点日時など）
- 異常系（不正値・欠損・解釈不能入力）
- 状態遷移（操作前後の変化・冪等性）
- データマッピング（入力モデル⇄内部表現⇄出力）

## ルール
- テストコードに確認項目からテスト観点までの詳細説明をコメントに残す
- 以下のようなテストケースの作成は禁止
  - private関数や内部構造への依存
  - 実装手順の検証（呼び出し回数など）
  - 過剰モック
  - UI／DOM／FW依存テスト
  - スナップショット中心の検証
- コード変更を行う場合、**必ず全テストがパスした状態で提出すること**
- テストを行うかどうかの確認は厳禁。**必ずテストを実行すること**
- テスト用コマンドに`--reporter=list`, `--max-failures=1`, `--headed`などのオプションを**勝手に追加しない**
- テスト結果確認の際に` | Select-Object -First N`などでエラー出力を**絶対に切り詰めない**

## 設計原則
- 1テスト＝1仕様ルール
- Given/When/Thenで意図を明確化
- 最小データで再現可能にする
- 結果のみを検証し、実装自由度を残す

## テスト品質基準
- 実装を変更してもテストが壊れないこと。
- 仕様変更時のみテストを更新する状態が理想。

## 実行手順
```bash
npm test
```

## 補足
- 無し

# E2Eテスト

## 目的
- ユーザー操作を起点に、システム全体が仕様どおり動作することを保証する。
- 機能単体ではなく「実際の利用シナリオが成立するか」を検証する。

## 対象範囲
- UI操作 → データ更新 → 再描画までの一連の流れ
- 画面とデータの整合性
- 外部I/O（Markdown読込・書込・状態反映）の結果
- ユーザーが知覚する挙動のみを検証対象とする

## 必須観点
- 主要ユーザーフロー（作成／編集／移動／更新）
- 状態反映タイミング（操作後に正しく反映されるか）
- データ永続化（再読込後も状態が維持されるか）
- 境界操作（端日時・空データ・大量表示）
- UI操作と内部データの最終一致

## 設計原則
- 「ユーザーが何をしたか」だけを書く
- 操作 → 結果 のみ検証（途中状態は見ない）
- セレクタは意味的属性を使用（role / text など）
- リトライではなく原因排除で安定化させる

## テスト品質基準
- ユーザー操作シナリオが常に再現可能で、
- 内部実装変更の影響を受けないこと。

## E2Eテスト実行手順

### 前提
- グローバルインストール禁止。
- すべて npx 経由で実行すること。

### テスト実行フロー
- 開発サーバ起動
```bash
npm run dev
```
- サーバ起動待機（自動待機必須・sleep禁止）
- テスト実行
```bash
npm run test:e2e
```
- 開発サーバが起動できない場合、またはプロセスが存在しないエラーが発生する場合
  - 数回リトライ
  - リトライも失敗する場合はテストイテレーション中止。報告を行い指示を待つ。
- 失敗時
  - 原因修正 → 再実行（テスト改変は禁止）

### MCP利用方針
- テスト時のみPlaywrightをオンデマンド実行
- ローカル環境へ依存物を常駐させない
- コード生成より「実ブラウザ検証」を優先

### 禁止運用
- npm install -g の使用
- 固定パス参照
- 手動ブラウザ確認による合格判定

### CI的運用ルール
- 「E2Eが通る状態＝出荷可能状態」とみなす。

## ⚠️ 追加注意事項（UI操作系テストの実装規約）

E2Eテストにおいて、UI操作を伴う機能（ドラッグ・リサイズ・スクロール連動・座標操作など）は 通常のDOMイベント検証とは異なる「ポインタ挙動シミュレーション」として扱う必要がある。

以下の規約を必ず遵守すること。

### ❌ 禁止事項（AIが誤生成しやすい実装）

- 内部関数の直接呼び出し
- DOM構造への過度な依存（class階層固定など）
- 見た目のみの検証（スクショ一致）
- wait固定秒数による同期
- テストのための実装変更
- **コンソールエラーを確認せずにテストを完了とすること**
- 以下の方法はユーザー操作の再現にならないため全面禁止とする。
  - `dispatchEvent()` による click / drag / drop 等の人工イベント発火
  - `new DragEvent()` / `new MouseEvent()` 等の手動生成イベント
  - `locator.dragTo()` / `page.dragAndDrop()` の使用
  - DOM状態を直接変更して操作を再現する行為
  - `evaluate` 内でUIイベントを発火させる行為
  - CSS変更・属性変更による「操作の代替」
  - スクリーン座標を無視した論理的移動
  - 「イベントが発火したこと」のみを検証するテスト
  - これらはブラウザ内部状態（hit-test / pointer capture / layout遷移）を経由しないため、E2Eとして無効。

### ✅ 必須実装（実ユーザー操作の再現）

ドラッグ・移動・リサイズなどのUI操作は  
必ずポインタ軌跡を用いて再現すること。

操作は次の低レベルAPIのみ使用する：
```javascript
page.mouse.move()
page.mouse.down()
page.mouse.move({ steps: N })
page.mouse.up()
```

### ✔ 軌跡生成ルール（必須）

DnD・スライド操作では、以下を必ず満たすこと：

- `move` は複数stepで分割（最低5step以上）
- 始点→終点を一度に移動してはならない
- 必ず途中座標を通過させること
- 人間の操作速度に近い移動を行うこと

### ✔ 同期の取り方（固定wait禁止の代替）

固定秒waitは禁止されているが、  
UIフレーム更新を跨がないテストも禁止とする。

同期は以下のいずれかで取る：

- 状態変化に対する `expect` のリトライ
- `requestAnimationFrame` を跨ぐ最小待機
- DOM反映の完了検知（描画・状態更新の終了）

時間ではなく状態で待機すること。

### ✔ 検証対象の原則

E2Eで検証するのは以下のみ：

- ユーザー操作の結果として最終状態が正しいか
- データとUIが一致しているか

検証してはならない：

- 中間DOM状態
- `transform` 値
- ピクセル位置
- 実装依存の内部構造

### ✔ セレクタ戦略

操作対象は以下の優先順位で取得する：

1. `role`
2. `aria-*` 属性
3. 意味的data属性（`data-testid` 等）
4. `text`

class名やDOM階層構造への依存は禁止。

### ✔ UI操作E2Eの対象範囲制限

ドラッグ・描画・スクロール等の操作は  
代表シナリオのみE2Eで保証する。

網羅的な座標パターン検証は禁止。  
詳細ロジックは機能テストで担保すること。

### ✔ テスト失敗時の原則

UI操作系テストが不安定な場合：

- retryで解決してはならない
- wait追加で誤魔化してはならない
- セレクタ強化で解決してはならない

必ず「操作再現方法がユーザー挙動と一致しているか」を見直すこと。

## 🎯 この規約の目的

E2EテストをDOMテストへ劣化させず、  
実際のユーザー操作を再現した統合検証として成立させることを目的とする。
