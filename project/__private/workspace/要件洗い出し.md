# 思想
- タスク管理にカレンダーを使いたい
- 巷のアプリおよびライブラリは自分の要望を満たさない
	- 使い方が限定的・固定化されている（fullcalendar）
	- 内部データや構造を秘匿していてカスタマイズができない（fullcalendar
	- そもそもカスタマイズできるような実装をしていない（決め打ち）
	- 有償（svar）
- 今後の自分のために、純粋なカレンダーのUI機能だけを提供する汎用性の高いカレンダーライブラリを作成したい。
	- 定義したデータ構造さえ受け取れれば即描画できる仕組みにする。利用側での作業はデータ作成＋データ受け渡しのみ。
	- 正のデータべースをカレンダー側で持つか利用側（外部）で持つか選択できるようにする。これで面倒なデータの2重管理を完全排除し、責任の境界を明確にする。
	- VSCODEやobsidianの拡張機能・ElectronやTauriに組み込めるような、環境を選ばない誰でも使えるライブラリを目指したい

# ライブラリの概要
- カレンダー用のデータを受け取り、それをUI表示する（HTML）
- typescriptベースで、svelteを使用
- ページ全体のUIの一部であるということを強く意識
	- 外のUIへの依存、および書き換えは行わない（疎結合）
	- このライブラリが１つの機能として独立

# データの定義
- カレンダーに存在する情報は、見た目が似ていても本質的に **2種類の異なる概念** を持つ。

| 種類        | 本質           | 完了概念 | 例                 |
| ----------- | -------------- | -------- | ------------------ |
| Task        | 成果を生む行為 | ✔ある    | 資料作成・レビュー |
| Appointment | 時間拘束       | ✖ない    | 会議・移動・立会い |

- なぜ分ける必要があるか。この2つを「Event」として混在させると、必ず以下の問題が発生する：
	- 予定に「進捗率」を持たせる矛盾
	- 完了チェックの意味が崩壊
	- Gantt表示時に予定がタスク扱いされる
	- 工数分析と稼働拘束分析が分離不能になる
	- Markdown記述ルールが曖昧化する
したがって本システムでは  
**Task と Appointment を異なるエンティティとして扱う。**


## 3. ドメインモデル全体像
```
CalendarItem（抽象概念）
 ├─ Task（実行対象）
 └─ Appointment（時間占有）
```

両者はUI上並んで表示されても、内部モデルは完全に分離する。

## 4. Task 定義（やるべきこと）

### 4.1 概念

「完了されることを前提とした作業単位」
成果物・責任・進捗を持つ。

### 4.2 使用シーン

- ToDo
- 開発作業
- ドキュメント作成
- 修正対応

### 4.3 データ構造

|             |               |      |                       |
| ----------- | ------------- | ---- | --------------------- |
| フィールド  | 型            | 必須 | 説明                  |
| id          | string        | ✔    | 一意識別子            |
| title       | string        | ✔    | タスク名              |
| start       | datetime      | 任意 | 開始予定              |
| due         | datetime      | 任意 | 期限                  |
| status      | enum          | ✔    | `todo / doing / done` |
| progress    | number(0-100) | 任意 | 進捗率                |
| assignee    | string        | 任意 | 担当者                |
| parentId    | string        | 任意 | 上位Task              |
| tags        | string[]      | 任意 | 分類                  |
| description | text          | 任意 | 詳細                  |

### 4.4 特徴

- 「end」ではなく **due（期限）** を持つ
- 完了状態を持つ
- 時間占有しない場合も存在できる

## 5. Appointment 定義（予定・拘束）

### 5.1 概念

「その時間に存在する必要がある状態」
成果を生まない可能性があっても成立する。

### 5.2 使用シーン

- 会議
- 来客対応
- 移動
- 休暇
- 施設予約

### 5.3 データ構造

|             |          |      |                |
| ----------- | -------- | ---- | -------------- |
| フィールド  | 型       | 必須 | 説明           |
| id          | string   | ✔    | 一意識別子     |
| title       | string   | ✔    | 予定名         |
| start       | datetime | ✔    | 開始時刻       |
| end         | datetime | ✔    | 終了時刻       |
| isAllDay    | boolean  | 任意 | 終日予定       |
| location    | string   | 任意 | 場所           |
| attendees   | string[] | 任意 | 参加者         |
| isBusy      | boolean  | ✔    | 時間占有フラグ |
| tags        | string[] | 任意 | 分類           |
| description | text     | 任意 | 詳細           |

### 5.4 特徴

- 完了概念を持たない
- progress / status を持たない
- 時間枠そのものが存在理由


# 見た目・操作性
- １日・１週間・月・年表示
	- 一番重要なのは１週間
- 10000件くらいの表示も耐えられる（SVGが候補か？）
- ズームインアウトでtickintervalを自由に変えられる（hammerjsのようなイメージ）
- DnDで日時の移動ができる。開始終了のみの編集もできる
- メトロカラーなどを使ったモダンな配色
- やりすぎない程度の丸みを帯びたデザイン（お堅い企業のお堅い部署でも受け入れられる程度）
- 目指すイメージはfullcalendarや、SVARのカレンダー

# 処理/実装方式
- 全体
	- とにかくカスタマイズの余地を残す。変更に強い仕様・実装にする。
	- 環境依存の実装はしない。なるべくjs/tsの標準環境のみで動くようにする。どうしても難しい場合は、何に依存しているのかを明確にする。
	- ライブラリの中で何かのオブジェクトにイベントを登録するような処理がある場合、イベント関数はインラインではなく登録前に別で定義し指定するようにする。また、そのイベント登録機能自体を外部に公開し、だれでもカスタマイズできるようにしておく
		- （例）カレンダーの●●をクリックしたときに発生するイベントをライブラリ利用者も登録できるようにしておく
- UI
	- １日・１週間・月・年表示
		- １日・１週間表示はvertical、月・年はhorizontal
		- まずは１週間表示を開発
	- 後からいつでも構造とスタイルを変更できるような実装方法にしておく
		- 主要なパーツはレンダリング関数を用意しておいて、上書きできるようにしておく
	- できるだけCSSにスタイリングさせ、JSでの重いスタイリングは避ける。
	- 思想に合致するCSSライブラリがあれば使ってOK。裏側でJSでゴリゴリに処理させるようなライブラリはNG
	- スタイリングの固定値はできるだけ相対値を使い、ズームやビューポートの変更に対応できるようにする
- データ処理
	- パフォーマンスに壊滅的なデメリットが出ない限り、データの持ち方は修正ではなく上書き。必ずデータの伝播の流れは一方通行にする（reduxのようなイメージ）。
	- 正のデータべース（オブジェクト）をカレンダー側で持つか利用側（外部）で持つか選択できるようにする。
		- 巷のライブラリでよくあるライブラリ側に正のデータを強要するパターンは絶対に避けたい。
		- そのために、データベースの操作は必ずイベントを通して操作する。このイベントは外部にも公開し、API的に利用する。
- その他
	- 日付計算は自分でやらず必ず専用ライブラリに任せることを徹底する（luxonなど）

























# カレンダーUIライブラリ仕様書

# 1. 設計思想（Why）

## 1.1 背景

既存ライブラリ（例：FullCalendar、SVAR）は以下の問題を抱える：

- 利用方法が固定的で拡張困難
- 内部データ構造がブラックボックス
- 「Event」概念に過度に集約されている
- 有償・商用制限が存在するケースがある
- 利用者にデータ管理方法を強要する設計が多い

本プロジェクトではこれらを回避し、

> **純粋な「描画専用カレンダーUIエンジン」を実装する**

ことを目的とする。


## 1.2 基本理念

### ① UIとデータ責務の分離

- カレンダーは「描画エンジン」
- データの正本（Single Source of Truth）は選択制
- 双方向データ同期は行わない
- データ変更は必ずイベント経由

### ② ドメインの厳密分離

「Task」と「Appointment」は絶対に統合しない。

### ③ 組み込み前提設計

- VSCode拡張
- Obsidian拡張
- Electronアプリ
- Tauriアプリ
- Webアプリ
いずれにも依存なく組み込める設計。

### ④ カレンダーライブラリとは

- 初めはfullcalendarを目指して作成する（[fullcalendar/fullcalendar: Full-sized drag & drop event calendar in JavaScript](https://github.com/fullcalendar/fullcalendar)）
- fullcalendarのソースコードは必ず確認してください

# 2. ライブラリ概要

## 2.1 役割

入力：

```
定義済みデータ構造（Task / Appointment）
```

出力：

```
レンダリングされたHTML UI
```

このライブラリはアプリ全体の一部であり、

- 外部UIを書き換えない
- グローバル状態を持たない
- DOM外部に副作用を出さない

## 2.2 技術スタック

- TypeScript
- Svelte（コンポーネントベース）
- 環境依存なし（Node専用API禁止）
- 日付計算は必ず外部ライブラリ利用（例：Luxon）

# 3. ドメインモデル

## 3.1 中核概念

```
CalendarItem（抽象）
 ├─ Task
 └─ Appointment
```

UI上は混在表示可能だが、  
**内部モデルは完全分離する。**

## 3.2 データ構造

| フィールド  | 型       | 必須 | 説明                              |
| ----------- | -------- | ---- | --------------------------------- |
| id          | string   | ✔    | 一意識別子                        |
| type        | string   | ✔    | 名称                              |
| title       | string   | ✔    | 名称                              |
| start       | datetime | 任意 | 開始予定                          |
| end         | datetime | 任意 | 期限                              |
| status      | enum     | ✔    | `todo / doing / done / undefined` |
| parentId    | string   | 任意 | 親Task                            |
| tags        | string[] | 任意 | 分類                              |
| description | text     | 任意 | 詳細                              |

# 4. Task 定義

## 4.1 概念

成果を前提とする作業単位。  
完了・進捗・責任を持つ。

## 4.2 設計上の重要事項
   
- 時間占有しないTaskも許容

# 5. Appointment 定義

## 5.1 概念

時間拘束そのもの。
成果の有無は問わない。

## 5.2 設計上の重要事項
   
- 完了概念を持たない

# 6. なぜ分離するのか（必須理解事項）

両者を「Event」として統合すると：

- 予定に進捗率を持たせる矛盾
- 完了概念の破綻
- 工数分析不能
- 稼働拘束分析不能

したがって：

> Task と Appointment は異なるエンティティである。

# 7. UI仕様

## 7.1 表示モード

| 表示  | 軸方向             |
| ----- | ------------------ |
| 1日   | Vertical           |
| 1週間 | Vertical（最優先） |
| 月    | Horizontal         |
| 年    | Horizontal         |

初期実装は「1週間表示」。

## 7.2 操作性

- DnDによる日時移動
- 開始・終了のみ変更可能
- ズームイン / ズームアウト
- tick interval変更可能
- 10,000件表示に耐える

描画方式候補：

- SVGベース
- 仮想スクロール
- 差分レンダリング

## 7.3 デザイン方針

- メトロ系配色
- 企業利用可能な控えめデザイン
- 過度な装飾禁止
- 角丸は適度
- スタイルは極力CSSへ委譲

# 8. アーキテクチャ

## 8.1 データフロー

一方向データフロー（Redux思想）

```
External Data
   ↓
Calendar
   ↓
Event Dispatch
   ↓
External Handler
   ↓
Data Update
   ↓
Re-render
```

---

## 8.2 データ管理モード

### モードA：外部管理型

- データ正本は利用側
- カレンダーは完全ステートレス

### モードB：内部管理型

- カレンダーが正本保持
- API経由で操作
必ず選択可能にする。

## 8.3 イベント設計

- すべての操作はイベント経由
- インライン関数禁止
- 登録関数は外部公開

例：

```
onItemClick
onItemMove
onItemResize
onViewChange
```

---

## 8.4 カスタマイズ設計

- 主要パーツはレンダリング関数として分離
- 上書き可能設計
- スタイル固定値は相対値優先
- CSS変数活用

# 9. 非機能要件

- 環境依存なし
- グローバル汚染禁止
- DOM直操作禁止
- 日付計算は専用ライブラリ必須
- パフォーマンス劣化時のみ最適化を許可


# 10. 絶対に守るべき原則

1. TaskとAppointmentを統合しな
2. データ正本を強要しない
3. 双方向データ同期をしない
4. 内部ロジックを秘匿しない
5. カスタマイズ可能性を常に残す
