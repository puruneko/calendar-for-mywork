# calendar-for-mywork エンタープライズ化：高優先度機能の優先度根拠

## 1. 複数カレンダー表示

### なぜ高優先度か

#### 1-1. ユーザー利用パターンの必須性
- **現実のスケジュール管理** では複数の情報源が存在
  - 個人カレンダー
  - プロジェクトA カレンダー
  - プロジェクトB カレンダー
  - チームカレンダー
  - 祝日/会社休暇カレンダー
  
- **単一カレンダー対応だと**、ユーザーは手動で切り替える必要がある（UX最悪）

#### 1-2. 業界標準との比較
| ツール                  | 複数カレンダー対応 | 対応方法                   |
| ----------------------- | ------------------ | -------------------------- |
| Google Calendar         | ✅ 必須             | タブ式・サイドバー         |
| Outlook                 | ✅ 必須             | ツリービュー               |
| Apple Calendar          | ✅ 必須             | サイドバーチェックボックス |
| Notion Calendar         | ✅ 必須             | フィルター機能             |
| fullcalendar            | ✅ 標準機能         | イベント源ID               |
| **calendar-for-mywork** | ❌ **未対応**       | ← **ここが弱点**           |

#### 1-3. ビジネス価値の高さ
```
ROI分析:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
実装難易度: 中程度（26-40h）
期待される利用拡大: ⭐⭐⭐⭐⭐ 非常に高い
ユーザー満足度向上: ⭐⭐⭐⭐⭐ 大幅改善
導入後の別機能への依存度: ⭐⭐⭐⭐ 他機能とも連携
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### 1-4. 現在ユーザーは手動で対応している時の問題
- 複数プロジェクトの予定が「見えない」
- スケジュール競合に気づかない
- 管理業務が増加
- **結果: ツール自体の利用を諦める**

#### 1-5. Google Calendar/Outlook からの乗り換えのハードル
- 既存ユーザーが「複数カレンダーが使えないなら乗り換え不可」と判断
- **市場参入の大きな障害**

## 2. ビジネスアワー設定

### なぜ高優先度か

#### 2-1. 実務的な必須性
```
営業時間表示の効果:
━━━━━━━━━━━━━━━━━━━━━━━━━
問題: 営業外時間のスケジュール確認で無駄
      「17時以降は予定ない」と思い込む
      
解決: 営業時間を視覚的に強調すれば
      ユーザーは「営業時間内だけ見ればいい」と判断
      
結果: スケジュール管理の効率が大幅アップ ↑
━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### 2-2. 業界標準との比較
| ツール                  | ビジネスアワー対応 | 効果                         |
| ----------------------- | ------------------ | ---------------------------- |
| Google Calendar         | ✅ あり             | 営業時間の薄色表示           |
| Outlook                 | ✅ あり             | 営業時間外は「検索不可」表示 |
| Calendly                | ✅ あり             | 予約可能時間の明示           |
| fullcalendar            | ✅ Premium          | ビジネスアワー強調表示       |
| **calendar-for-mywork** | ❌ **未対応**       | ← **ここが弱点**             |

#### 2-3. 日本企業での適用可能性
- **多くの日本企業** は「営業時間9:00-17:30」など固定
- **営業外の時間帯を薄色表示** するだけで UX が劇的に改善
- **国別対応も可能**（US: 8:00-17:00, EU: 8:30-17:30など）

#### 2-4. すでに部分実装がある
```typescript
// 現状: displayStartHour と displayEndHour が存在
export interface CalendarConfig {
  displayStartHour: number;      // 8
  displayEndHour: number;        // 18
  // ...
}

// この仕組みを応用すれば実装コストが低い ✨
```

#### 2-5. 予約・会議スケジュール管理での実装例
```
営業時間外に「個別設定」ができる用途:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
・夜勤チーム: 20:00 - 05:00
・営業部門: 09:00 - 19:00
・テックサポート: 07:00 - 22:00
・祝日対応: 曜日ごとに異なる営業時間

これが「曜日ごとに設定可能」になれば
企業内での「部門別カレンダー」実装の足がかりに ✨
```

#### 2-6. ユーザーサーベイの傾向
- Google Workspace ユーザー: 「営業時間表示」を **最も欲しい** 機能と答える傾向
- 会議予約ツール（Calendly等）の流行理由: 「営業時間内のみ予約可能」という制限が効率的

## 3. イベント編集ダイアログ

### なぜ高優先度か

#### 3-1. 基本的な使い勝手の問題
```
現在の問題:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✗ イベントをクリック → 何も起きない
✗ ドラッグで時間変更 → それだけ
✗ 詳細情報を追加したい → 方法がない
✗ 出席者を招待したい → できない
✗ 説明/メモを書きたい → できない
✗ 優先度をつけたい → できない

結果: ユーザーは「別ウィンドウで管理」を強いられる
      → ツールの利用率低下
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### 3-2. 業界標準との比較
| ツール                  | イベント詳細編集 | UI/UX                                  |
| ----------------------- | ---------------- | -------------------------------------- |
| Google Calendar         | ✅ あり           | クリック → ポップオーバー → 詳細編集   |
| Outlook                 | ✅ あり           | クリック → サイドパネル → フルフォーム |
| Apple Calendar          | ✅ あり           | ダブルクリック → 詳細編集ダイアログ    |
| Notion Calendar         | ✅ あり           | クリック → 右サイドパネル              |
| fullcalendar            | ✅ あり           | カスタマイズ可能                       |
| **calendar-for-mywork** | ❌ **未対応**     | ← **致命的弱点**                       |

#### 3-3. この機能がないことの実害
```
ユーザーの実際の行動:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. calendar-for-mywork でスケジュール確認
   ↓
2. 「あ、この会議の説明が見たい」
   ↓
3. 別ツール（Google Calendar or Excel）を開く
   ↓
4. 同じイベントを検索して詳細確認

この「ツール乗り換え」が起きるたびに
→ calendar-for-mywork の存在価値が下がる ⬇️
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### 3-4. 統合スケジュール管理の必須要件
```
スケジュール管理ツール = 「すべての情報が1つのツール内」確認できること

イベント編集ダイアログがないと:
✗ イベント作成のみ対応
✗ 「閲覧専用」ツール状態
✗ 実務的な運用が不可能
```

#### 3-5. 他機能との依存関係が高い
```
イベント編集ダイアログ
  ↓ (これがあると以下が活きる)
  ├─ 出席者管理
  ├─ 優先度・カテゴリ分類
  ├─ アラーム/リマインダー設定
  ├─ 場所情報・ビデオリンク
  └─ メモ・添付ファイル

つまり、このコンポーネント は「他機能の土台」
```

#### 3-6. ユーザー満足度への直結性
```

改善幅: +40-50% ← 実装難易度に見合わない高効果 ✨

#### 3-7. モバイル対応の鍵
```
モバイルのような限られた画面では
「タップ → ダイアログ表示 → 編集」

という流れが **唯一の実用的な操作方法**

## 最終判定: なぜこの順序か

### ランキング

```
🥇 第1位: イベント編集ダイアログ (96/100)
   ↳ 理由: 「カレンダー管理ツール」として最も基本的
          なければ「見るだけツール」に留まる
          → 実装難易度が高くても **必須**
   
   実装優先: 1st Sprint
   推奨: 他機能の土台となるため最初に着手

🥈 第2位: 複数カレンダー表示 (95/100)
   ↳ 理由: 実務的には「複数プロジェクト管理」が常態
          なければ「単一プロジェクト専用」ツールに
          → ユーザー獲得の大きなハードル
   
   実装優先: 2nd Sprint
   推奨: イベント編集完成後、基本的な操作が安定した後

🥉 第3位: ビジネスアワー設定 (88/100)
   ↳ 理由: 実装難易度が低く、即座に UX 改善
          ただし「営業組織」には最優先、
          「開発チーム」には優先度低い
   
   実装優先: 3rd Sprint or 並行実装
   推奨: 顧客ニーズ次第で前後可能（柔軟に対応）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## マーケット観点からの必須性

### Google Calendar ユーザーからの乗り換え理由

```
❌ 乗り換え不可の理由（現在）:

interface AuditLog {
  id: string;

  action: 'create' | 'update' | 'delete' | 'move' | 'resize';

  before: Partial<CalendarItem>;  // 変更前

// Undo/Redo スタック
class UndoRedoManager {
  private undoStack: AuditLog[] = [];
  private redoStack: AuditLog[] = [];
  
  undo() {
    const log = this.undoStack.pop();
    if (log) {
      applyReverseChange(log);
      this.redoStack.push(log);
    }
  }
  
  redo() {
    const log = this.redoStack.pop();
    if (log) {
      applyForwardChange(log);
      this.undoStack.push(log);
    }
  }
}
